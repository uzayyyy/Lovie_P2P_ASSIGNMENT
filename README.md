# Lovie Payment Request Assignment

P2P payment request prototype built for the Lovie interview workflow using Spec-Kit, React Admin v5, TypeScript, and Supabase.

## Live demo

| Path | URL |
|------|-----|
| Interactive demo (no login) | **https://lovie-p2p-demo.vercel.app/demo** |
| Public share-link preview | **https://lovie-p2p-demo.vercel.app/request/demo-public-pending** |
| Live Supabase-backed app | **https://lovie-p2p-demo.vercel.app/** |

> The `/demo` route is a public sandbox — no email or account required.  
> Use it to exercise the full create → pay / cancel flow instantly.

## Project overview

The feature lets a user request money from a friend using email or phone, manage incoming and outgoing requests, inspect request details, simulate payment fulfillment, cancel or decline pending requests, and handle seven-day expiration.

The implementation is split into:

- A production path using React Admin + Supabase auth/data persistence.
- A public demo sandbox for recruiter-friendly evaluation and stable E2E automation.

## Features shipped

- Request creation with recipient email or phone, amount, optional note
- Validation for contact format, positive amount, note length, and self-request blocking
- Unique request IDs and shareable links
- Dashboard with incoming and outgoing requests
- Statuses: Pending, Paid, Declined, Cancelled, Expired
- Detail view with amount, note, sender/recipient info, timestamps, countdown
- Incoming actions: Pay, Decline
- Outgoing action: Cancel
- 2.5-second simulated payment processing
- Public share-link preview route
- Demo sandbox route for end-to-end review without auth

## Database setup

For a clean Supabase project, use either:

- migration chain in `src/payment-request/supabase/migrations/`
- one-shot SQL in `src/payment-request/supabase/setup.sql`

The schema includes:

- `profiles`
- `payment_requests`
- RLS policies for sender/recipient access
- recipient resolution trigger
- public view for shareable links
- hourly expiration SQL for `pg_cron`

## Notes and tradeoffs

- The live app uses Supabase auth and persistence.
- The public `/demo` sandbox exists to make the assignment testable without inbox access.
- Phone-based requests are supported for creation and display; email auth remains the primary authenticated path for the live app.
